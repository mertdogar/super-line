# super-line

Strictly-typed realtime data bus for TypeScript — one contract for every pattern on the wire (requests · events · subscriptions), end-to-end types, no codegen. Same API on one node or a cluster. pnpm monorepo.

## Commands (run from repo ROOT)

```bash
pnpm test         # the REAL suite: fast lane (parallel) + integration lane. `pnpm -r test` runs NOTHING.
pnpm test:fast    # fast lane only — no Docker, parallel forks; the inner loop
pnpm test:integration  # heavy lane — Docker brokers (one shared redis+rabbitmq via global-docker.ts), zeromq/libp2p sockets, serial
pnpm typecheck    # root tsc — the REAL check. `pnpm -r typecheck` only covers 2 of 18 projects.
pnpm lint         # oxlint (NOT eslint)
pnpm build        # pnpm -r build (tsup)
pnpm docs:dev     # VitePress dev server
pnpm docs:build
```

## Layout

`packages/{core,server,client,react,control-center}` · `packages/transport-{websocket,http,libp2p,loopback}` (pluggable client↔server transports) · `packages/adapter-{redis,libp2p,rabbitmq,zeromq}` (server↔server fan-out) · `packages/collections-{memory,sqlite,pglite}` (typed row collections — `CollectionStore` backends) · `packages/collections-crdt-{memory,libsql,pglite}` (CRDT document collections — `CrdtCollectionStore` backends; `-pglite` is the `self`/Electric tier) · `packages/tanstack-db` (TanStack DB sync adapter) · `packages/plugin-{inspector,auth}` (paired plugins: Control Center inspector · first-party authentication) · `examples/` · `docs/` (VitePress) · `skills/super-line/` (published agent guide for *users* of super-line, not dev-context for this repo).

- `defineContract` lives in `@super-line/core`; `createSuperLineServer` in `/server`, `createSuperLineClient` in `/client`, `createSuperLineHooks` in `/react`. The wire error type is `SuperLineError`.
- **Client↔server transport is pluggable.** Server takes `transports: [...]`, client takes `transport:`; `authenticate` receives a normalized `Handshake`. `webSocketServerTransport`/`webSocketClientTransport` (default) live in `@super-line/transport-websocket`; HTTP-SSE/long-poll in `/transport-http`; libp2p in `/transport-libp2p` (bring-your-own node); in-memory `/transport-loopback` for tests. Transport interfaces (`RawConn`/`ServerTransport`/`ClientTransport`/`Handshake`) are in `@super-line/core`. Migration notes: `TRANSPORT-MIGRATION-NOTES.md` at repo root.
- `docs/reference/` is typedoc-GENERATED — never hand-edit; it regenerates from source on `docs:build`.

## Gotchas

- **Tests and typecheck only work from root**, not via `pnpm -r` (no package has a `test` script; only 2 have `typecheck`). `pnpm -r test` silently passes while running nothing.
- `packages/server/test/reconnect.integration.test.ts` → "auto-reconnects after an abrupt drop" is timing-flaky under full parallel load; it passes in isolation. Don't chase it as a regression. (It lives in the serial integration lane for exactly this reason.)
- The lane split is the `heavy` list in `vitest.config.ts` (Docker/socket/timing-sensitive files), NOT the `.integration.test.ts` suffix — most integration-named files are loopback tests and run in the parallel fast lane. To run one heavy file: `pnpm test:integration <name>`.
- ESM-only packages. Lint/format is oxlint/oxfmt, not ESLint/Prettier.
- **A public-surface change isn't done until `skills/super-line/` is updated** — all four files (`SKILL.md` + `REFERENCE.md` + `RECIPES.md` **and** `AGENTS.md`, which drifts because it's easy to forget). It's the guide consumers' agents run on, so a stale one makes them write broken code. **Never cite an ADR or a `PLAN-*.md` in it** — those are repo-internal and not published; state the behavior instead.
- **A package NEVER hard-depends on a sibling `@super-line/*`.** Internal deps are `peerDependencies` with an explicit wide range (`">=<x.y.0> <1.0.0"`, hand-written — *not* `workspace:^`) plus a `devDependencies: workspace:^` mirror that does the local linking. A regular dep publishes as a 0.x caret, so the next core minor installs two physical copies of core and `instanceof SuperLineError` silently returns false (typed errors degrade to `INTERNAL`). Enforced at publish by `scripts/check-manifest.mjs`, which also keeps `packages/core/src/version.ts` in step with core's `package.json`.

## Architecture

One contract is the single source of truth (`defineContract`), imported by both server and client, split by direction (`clientToServer`/`serverToClient`) and scoped by role (a `shared` base + per-role blocks). **Server-authoritative**: the server owns rooms/topics, authorizes subscribes, and validates every inbound message; roles are fixed at connect by `authenticate(handshake)` (cross-role calls → `NOT_FOUND`). Three wire patterns: requests (req/res), events (server push), topics (client subscribe). The **client↔server wire is a pluggable transport** (WS/HTTP/libp2p/loopback — see Layout). Separately, **server↔server cross-node fan-out** goes through a pluggable `Adapter` (in-memory default; Redis/libp2p/rabbitmq/zeromq adapters ship) plus a server-side cluster bus (`server.publish`/`server.subscribe` with local echo). Don't confuse the two: transports carry client↔server bytes; adapters carry node↔node fan-out (but `clustering: 'self'` collection backends own a central Postgres + Electric replica and bypass the adapter entirely).

**Collections subsystem (ADR-0006).** The relational successor to the LWW store family: typed rows declared **on the contract** (`collections: { name: { schema, key, references? } }` in `defineContract`), so the server validates every write and types flow end-to-end (`RowOf<C,N>`). super-line is the server-authoritative **sync source**; **TanStack DB is the client query engine** (joins/live-queries/optimism) via the `@super-line/tanstack-db` adapter — there is no native query engine. Core owns a query **IR** + one evaluator (`packages/core/src/query.ts`: `evalExpr`/`applyQuery`/`matchesFilter` + `and`/`or`/`not`/`eq`/`in`/`like`… builders and `and`/`orFilters`) shared by change-routing, scan snapshots, and client re-filtering. Wire frames: `csub`/`cuns`/`cbat`/`cchg`. One `CollectionStore` backend per server serves all collections (single tx domain → atomic cross-collection batches); `collections-memory`/`-sqlite` are `relay`, `collections-pglite` is `self` (central Postgres + Electric→PGlite). **Typed tables (PLAN-collections-typed-tables.md):** the SQL backends store each LWW collection in its own table (`col_<name>` / `tablePrefix`) with real columns derived from the Zod schema via core's `planColumns` (scalars → columns; records/unions/nested/optional+nullable → per-field JSON columns; non-Zod → key + `_sl_data`); factories REQUIRE the contract's post-merge `collections` map; `<prefix>meta`/`col_meta` fingerprints gate drift (additive optional columns auto-ALTER, else refuse to boot); sqlite pushes exactly-compilable queries fully to SQL (JS evaluator stays authoritative otherwise; text ranges/order never pushed — UTF-8 vs UTF-16 order); pglite runs one Electric shape + `live.changes` per table and re-reads the full row from the local replica on partial UPDATE diffs so `RowChange.next` stays whole (construction DDL serialized via `pg_advisory_xact_lock`). **Row security** = server-side `policies: { ns: { read(principal,ctx)→IR filter | undefined, write(principal,op,next,prev)→bool } }`, deny-by-default (omit `read`/`write` ⇒ denied). Routing is stateless per-connection beyond the predicate (deliver if pre- OR post-op row matches → client re-filters; prev-less `self` deletes broadcast to all subs). **Advisory FKs**: `references` metadata + opt-in `checkReferences` server flag (no cascades). `srv.collection(n)` co-writes (policy-free, schema-validated); `client.collection(n).subscribe(query)` → non-optimistic live row-set + `insert`/`update`/`delete`/`batch`; react `useCollection`; inspector `listCollections`/`queryCollection` + Control Center Collections view. Client contract: **await `sub.ready` before depending on live delivery** (frames process concurrently). Design + decisions: `PLAN-collections.md`, `docs/adr/0006-collections-are-on-contract-typed-rows.md`; guide `docs/guide/collections.md`; example `examples/collections`.

**CRDT document collections (ADR-0007).** The CRDT doc family folds INTO collections — one `collection(n)` concept, two consistency models (LWW rows · CRDT docs). A CRDT collection is declared on the contract by a `crdt` key (`collections: { scenes: { schema, crdt: DocOptions } }`, no `key` — id is external); it is **opened by id, not queried** (whole-doc merge). The headline reversal of ADR-0003/0006: opaque CRDT deltas ARE now validated — **validate-before-commit** at the ingress node (the backend merges the delta onto a scratch copy, snapshots to plaintext, the server validates against the contract schema, then commits + fans out only if valid; relay nodes trust already-validated deltas). Reject → the writer resyncs (server-push resync deferred — see PLAN). Access = guard-shaped `CrdtCollectionPolicy` (`read(principal,id,snapshot?)→bool`, `write(principal,id)→bool`, deny-by-default), NOT the RLS filter shape. **Creation is server-authoritative** (`srv.collection(n).create(id,data)`; clients open existing docs; nonexistent → `NOT_FOUND`). Backend seam: `CrdtCollectionStore` (keyed by `(collection,id)`, no ACL, validate hook) — a SEPARATE backend from `CollectionStore` (`crdtCollections:` server option; CRDT never joins a cross-collection atomic batch). Client: `client.collection(n).open(id)` → `DocHandle` (relocated `ResourceReplica` surface); needs `crdtCollections: crdtCollectionsClient()` (the universal client engine); react `useDoc`. Wire frames: `cdopen`/`cdwr`/`cdchg`/`cddel`/`cdclose`, per-doc channel `d:<n>:<id>`. Packages: `collections-crdt-memory` (relay, + the universal `crdtCollectionsClient`), `collections-crdt-libsql` (durable·relay, `await crdtLibsqlCollections`, snapshot-per-doc), `collections-crdt-pglite` (**self**: `await crdtPgliteCollections` — central Postgres Yjs op-log + per-node Electric→PGlite replica, validate-before-commit at the ingress node before op-log append, no adapter). Inspector: CRDT collections surface in `listCollections` (synthetic `id` key) + `queryCollection` synthesizes `{id, ...snapshot}` doc-rows, so the Control Center Collections view browses them with no CC change. **Store family retired (Phase 3b, ADR-0007):** the `store(n)` API and all `store-*` packages are deleted — `removeAtPath`/`StoreChange`/`ResourceReplica` survive in `core/src/store.ts` only because the CRDT client `DocHandle` reuses them. CRDT-doc schemas must be **presence-tolerant** (`.catch`/`.optional`) — validate-before-commit can't require a field a concurrent merge may transiently drop (ADR-0008). Design: `PLAN-collections-crdt.md`, `docs/adr/0007-crdt-docs-are-typed-collections.md`, `docs/adr/0008-crdt-validation-is-scoped-to-present-values.md`.

**Contract-fragment plugins + auth.** ADR-0005's runtime-only plugin boundary grew a **contract-time half**: `defineContract({ plugins: [contractPlugin] })` merges a plugin's collections/roles/shared surface INTO the contract via a plain **intersection** (`ResolveContract`), so `RowOf`/`client.collection`/per-role `Requests` infer from the single materialized contract with zero type-threading — existing callers untouched (`defineContract` is overloaded; no-plugins = identity). Server-side, a plugin can also contribute `policies` (merged), and `SubtractHandlers` now drops **fully-plugin-owned** blocks to optional (so a host needn't pass empty `shared:{}`/`guest:{}`). Built on this: **`@super-line/plugin-auth`** (subpath package: `.` contract · `/server` · `/client` · `/react`) — first-party auth = email/password (scrypt), reusable hashed **access tokens**, per-connection durable **sessions**, **data-driven roles** (`roles[]` on the user row; only `guest` hardcoded), **API keys** (`slp_`, fixed role), **JWT** (jose HS256, server-minted signed/sealed assertions + `?jwt=` connect), safe `userPresence`, revocation-and-disconnect, and password reset via a host callback. Profiles and credentials are separate imperative surfaces, so API-key-only agents have no empty credential row. Wiring requires `plugins:[authContract()]` on the contract and a stable `nodeKey`, `authenticate: authKit.authenticate` + `identify` + `plugins:[authKit.plugin]` on the server, and `createAuth()`/`authClient()` on the client. Design: `PLAN-auth-connection-sessions-presence.md`; guide `docs/how-to/plugin-auth.md`; examples `examples/auth` + `examples/collections-chat`.
